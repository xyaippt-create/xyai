## 问题归因

归因：前台链路证据不足 + 多图队列目录快照不够稳。

具体表现：

1. `POST /api/output/select-popup` 已能返回本地文件夹，但前台此前没有在选择后显式触发 `/api/output/validate`，导致无法从浏览器控制台确认验证请求与返回。
2. XHR 上传此前只在 `appliedOutputDir` 非空时追加 `output_dir`，默认目录状态下 FormData 可能不带 `output_dir`。
3. 多图串行上传此前每张图处理时临时读取当前输出目录状态，理论上存在队列运行期间目录状态变化导致不同图片目录不一致的风险。
4. 后端已有 `resolve_output_dir(output_dir)`，会优先使用 request form 中的 `output_dir`，但缺少明确的控制台链路日志。

## 当前 output_dir 链路状态

### 1. `/api/output/validate`

状态：已补前台真实触发。

触发位置：

```text
用户点击“更换文件夹”
→ 后端弹出本地文件夹选择器
→ 前台收到 output_dir
→ 前台立即 POST /api/output/validate
→ 验证通过后更新 appliedOutputDir
```

控制台日志：

```text
[VisualMasterPro output_dir] request
[VisualMasterPro output_dir] response
```

### 2. `/api/upload` XHR FormData

状态：已补强。

每次上传都会记录：

```text
[VisualMasterPro output_dir] upload FormData
```

字段包括：

```text
file
mode
scale
format
output_profile
output_format
output_dir
```

### 3. 多图队列串行上传

状态：已补强。

队列启动时会固定一次输出目录：

```text
const queueOutputDir = resolveOutputDirForRequest()
```

随后每张图片都会使用这个 `queueOutputDir` 传入 XHR FormData，避免多图队列中途丢失或混用目录。

### 4. 后端 `/api/upload`

状态：代码检查通过。

后端当前逻辑：

```text
output_dir: str | None = Form(None)
output_root, output_dir_meta = resolve_output_dir(output_dir)
```

`resolve_output_dir()` 的优先级：

```text
request output_dir
→ last_used
→ default
→ runtime_fallback
```

当 request output_dir 有效时，任务元信息应为：

```json
{
  "output_dir_source": "request",
  "used_custom_output_dir": true
}
```

### 5. `task_result` 输出字段

状态：代码检查通过。

任务完成后 `update_task_result_output_meta()` 会写入：

```text
output_dir
output_path
output_filename
output_size
final_output_url
preview_output_url
used_custom_output_dir
output_dir_source
output_dir_created
output_path_exists
```

满足本轮要求。

### 6. `final_output_url` 预览

状态：代码检查通过，真实预览待复测。

前台对比页继续使用网络 URL，不使用本地盘符：

```text
http://localhost:8787/api/file/outputs/{filename}
```

后端 `resolve_public_file("outputs", filename)` 会优先读取 `output_file_index`，并回退搜索：

```text
正式输出目录
last_output_dir
default_output_dir
```

因此自定义输出目录中的文件理论上仍可通过 `/api/file/outputs/...` 预览。
